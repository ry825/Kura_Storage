## 現行リモートアクセス方針

Phase 1は、KuraStorage外で管理されるZeroTierをリモートアクセス方式とする。KuraStorageはZeroTier Networkの作成、Member認可、Managed IP割り当て、Node Identity、接続・切断を実行せず、OSまたは別アプリが確立したZeroTier経路をHTTPS通信路として利用する。

- DDNS、Router Port Forward、自宅Global IP公開、KuraStorage用Public受信Portを使用しない。
- Device登録はLocal Direct限定、登録済みDeviceの利用はLocal DirectまたはZeroTier経路で許可する。
- KuraStorageのUser、Device、Session、ファイル認可とZeroTier Member認可は、独立した防御境界とする。
- 環境固有値はGit管理外の`docs/environment-info.md`へ保存し、公開項目定義は`docs/environment-info.example.md`を参照する。
- ZeroTierの運用手順は本書と`docs/architecture-design.md`の管理境界に従い、詳細な手順書は運用作業時に`docs/operations/`へ追加する。

将来、自己管理WireGuard、VPSを介したVPN、または別の安全なオーバーレイネットワークへ変更する可能性がある。変更時は別途設計・脅威分析を行い、HTTPS、TLS検証、KuraStorage認証・認可を維持する。これらの代替方式はPhase 1の実装範囲に含めない。

## 1. プロダクト概要

### 1.1 名称

**KuraStorage** — 家族専用のローカルファイル管理・自動バックアップシステム

### 1.2 プロダクトコンセプト

- **家庭内に閉じたプライベートストレージ**  
  ファイル管理サーバーをRaspberry Piと接続HDD上で稼働させ、一般のインターネットには公開しない。

- **家族が迷わず使えるファイル管理**  
  Androidアプリ,Webブラウザから写真、動画、音声、PDF、テキストなどを見やすく閲覧・管理できる。複雑なサーバー操作を意識させない。

- **許可された接続で自動バックアップ**  
  Android端末で指定したフォルダについて、Local Direct、または信頼済み外部Wi-FiとZeroTierの双方が利用可能な場合だけ、新規・変更ファイルを自動アップロードする。

- **個人領域と家族共有を安全に分離**  
  ユーザーごとに閲覧可能なファイルを分離し、所有者が共有したファイルまたはフォルダだけを他の家族が閲覧・編集できるようにする。

- **接続環境を問わず表示品質を選択**  
  ローカル直接接続、外部Wi-Fi＋ZeroTier、モバイル通信＋ZeroTierなどの接続環境にかかわらず、写真・動画を低画質、中画質、元画質から選択して閲覧できる。バックエンドは選択された品質のデータだけをクライアントへ送信する。通信環境は初期画質の決定にのみ利用し、選択可能な品質を制限しない。自動バックアップは、ユーザーが許可したWi-Fi条件を満たす場合だけ実行する。

### 1.3 プロダクトビジョン

家族が写真、動画、書類などを外部クラウドへ預けることなく、自宅のストレージへ安全に保存し、スマートフォンやブラウザから自然に利用できる環境を提供する。

利用者はRaspberry Pi、HDD、ZeroTier、ファイルパスなどの技術的な仕組みを意識せず、「自分のファイルを見る」「家族と共有する」「自動でバックアップする」という日常的な操作だけで利用できる。

### 1.4 解決する課題

現在のFile BrowserやSyncthing系の構成には、次の課題がある。

- スマートフォンでの操作性や閲覧性が十分ではない
- ファイルの種類に応じたプレビューや再生機能が弱い
- ユーザー別の個人領域と家族共有を直感的に管理しにくい
- ZeroTier接続状態、サーバー接続状態、ストレージ状態が分かりにくい
- ファイルの検索、最近使った項目、同期状況などを一つの画面で把握しにくい
- 登録済みKuraStorageバックエンドへZeroTierを介さず直接到達できるローカルネットワークへ接続したときだけ、省電力で自動バックアップしたい
- 既存ツールを複数組み合わせることで、操作や管理が分散している
- モバイル通信や外部Wi-Fi経由で元画質の写真・動画を閲覧すると通信量が大きい
- ZeroTier経路で接続中であっても、モバイル通信や許可していないWi-Fiでは自動バックアップを開始したくない

### 1.5 提供する価値

- 家族のファイルを外部クラウドへ公開せず、自宅内で管理できる
- Androidアプリ、Webブラウザからファイル閲覧、再生、編集、整理を完結できる
- ローカル直接接続可能なWi-Fi、または許可済み外部Wi-Fi＋ZeroTierの条件を満たした際、写真などの差分だけを自動バックアップできる
- 家族ごとの個人ファイルと共有ファイルを明確に分けられる
- サーバーやHDDに問題がある場合、利用者が原因を理解できる表示を行える
- 将来的に同じバックエンドを利用したWebアプリを追加できる
- 低画質・中画質プレビューにより、接続環境を問わず通信量や表示速度を利用者が制御できる
- 登録済みWi-FiとZeroTier経路状態に基づき、自動バックアップを実行するネットワークを利用者が選べる

### 1.6 プロダクト原則

1. **使いやすさを最優先する**
2. **一般のインターネットへKuraStorage APIを直接公開しない**
3. **ZeroTier接続、ユーザー認証、端末・セッション状態、ファイル認可を別々の防御として扱う**
4. **保護APIは既定で拒否し、認証・認可に成功した要求だけを処理する**
5. **HDD上の実ファイルをファイル存在・内容の正とする**
6. **DBは認証、権限、検索、同期状態、Device、SessionなどKuraStorage自身の管理情報に使用する**
7. **端末側の削除をサーバー側へ自動反映しない**
8. **常時監視よりも省電力性と信頼性を優先する**
9. **利用者にLinuxの物理パスや内部構成を見せない**
10. **低・中画質データは元ファイルではなく、再生成可能な一時キャッシュとして扱う**
11. **Wi-Fi登録は自動実行ポリシーとして扱い、認証やサーバー本人確認の代替にしない**
12. **新規Android端末の登録はローカル直接接続からの初回ログイン時だけ許可する**
13. **Device・Session失効とZeroTier Member失効は別々に実行し、一方をもう一方の代替にしない**

### 1.7 接続経路の定義

本書では、曖昧な「家庭内LAN」または「同じネットワーク」という表現を、次の接続状態へ置き換えて扱う。

- **ローカル直接接続（`LOCAL_DIRECT`）**  
  Android端末またはWebクライアントとKuraStorageサーバーが、管理者が設定した同一IPサブネットに属し、ZeroTier経路を介さずKuraStorage HTTPSエンドポイントへ直接到達できる状態。同一サブネット確認、基盤ネットワークへの明示バインド、TLS証明書・ホスト名検証、期待するAPI応答の確認を必須とする。
- **安全なリモート接続（`REMOTE_SECURE`）**  
  Phase 1では、OSまたは別アプリが確立したZeroTier経路を通じてKuraStorage HTTPSエンドポイントへ到達でき、TLS証明書・ホスト名検証と期待するAPI応答の確認に成功した状態。状態名は将来の接続方式変更に依存させない。
- **未接続（`DISCONNECTED`）**  
  ローカル直接接続とZeroTier接続のどちらでも、KuraStorage HTTPSエンドポイントへ到達できない状態。

SSIDやBSSIDは`LOCAL_DIRECT`の判定条件に使用しない。異なるIPサブネットから到達できる場合は`LOCAL_DIRECT`として扱わず、Phase 1ではZeroTier経路の`REMOTE_SECURE`として扱う。

バックエンドプロセスがポートまたはUNIXソケットで待ち受けていることだけでは、クライアントから到達可能とはみなさない。リバースプロキシを含むクライアントからAPIまでの経路全体が利用可能であることを確認する。また、接続経路、ユーザー認証、Device・Session状態、HDD状態は別々に管理し、ZeroTierへ接続できてもログイン済みとはみなさない。

## 2. プロダクト範囲

### 2.1 Phase 1: MVP

以下を最初の正式完成範囲とする。

- Raspberry Pi上の.NET 10 API、Admin CLI、PostgreSQL 17、Nginx、専用HDD
- Android 10以上のネイティブアプリ
- Local Directまたは外部管理ZeroTier経由のHTTPS接続
- Local Direct限定の初回Device登録と、登録済みDeviceのLogin・Token更新・Logout・失効
- Admin CLIによるUser作成、Device一覧、Device失効、Security Lock解除
- Argon2idパスワード保護、Access Token、Refresh Token Rotation、監査ログ
- Userごとに分離された個人領域、File・Folder一覧、詳細、Folder作成
- Android SAFからの手動Streaming UploadとRange Download
- File・Folderのゴミ箱移動と元の場所への復元
- HDD Mount・storageId・Pathの検証、不完全Fileの非公開、最小の`FileOperation`復旧
- Server・Android自動Test、Raspberry Pi・Android実機E2E、Release APK、再現可能な配置手順

### 2.2 MVP後の機能追加

自動バックアップ、共有、検索・最近使用、名前変更・移動、完全削除・保持期限、分割Upload、アプリ内Media表示、サムネイル・低中画質生成、動画変換、テキスト編集、外部変更監視・`MISSING`判定は、MVP完成後に機能単位の別Steeringで計画・実装する。

### 2.3 Phase 2

- ブラウザから利用するWebアプリ
- Webからのファイル整理、検索、共有管理、テキスト編集
- AndroidとWebで同一バックエンドAPIを利用

### 2.4 Phase 3以降

- 高度な全文検索
- OCR
- 写真・動画の高度な分類
- ファイルバージョン管理の拡張
- 重複ファイル検出
- オフライン閲覧キャッシュ
- iOSクライアント

## 3. ターゲットユーザー

### 3.1 プライマリーペルソナ: システム管理者兼利用者

- 家庭内にRaspberry PiとHDDを設置している
- 家族の写真、動画、書類をローカルに保存したい
- 外部クラウドへ依存したくない
- サーバー、ZeroTier、ネットワークの基礎知識がある
- 家族には技術的な操作を要求したくない
- Androidアプリでは通常利用、共有設定、障害状態の確認を行う

**現在の課題**

- File BrowserやSyncthingなど、複数のツールを使い分ける必要がある
- スマートフォン向けUIが目的に合っていない
- 家族ごとの権限や共有設定を簡単に管理できない
- HDD、ZeroTier、同期の状態を横断的に確認しにくい
- 高権限の管理操作をネットワーク経由の管理画面へ公開したくない

**期待する解決策**

- 家族向けの一貫したAndroidアプリ
- 管理対象のファイルを安全に扱う共通バックエンド
- 高権限操作をRaspberry Pi上だけで実行できる管理CLI
- 家族の端末がローカル直接接続から初回ログインした際の自動Device登録
- 家族の端末が許可された接続条件を満たした際の自動バックアップ
- 問題発生時に原因を判別できる状態表示と監査ログ

### 3.2 セカンダリーペルソナ: 一般の家族ユーザー

- サーバーやZeroTierについて詳しくない
- スマートフォンで写真や動画をよく利用する
- 自分のファイルと家族共有ファイルを見たい
- 必要なファイルを簡単に検索、閲覧、ダウンロードしたい
- バックアップ操作を毎回手動で行いたくない

**期待する解決策**

- 接続できているか一目で分かる
- 数回の操作で写真や書類を開ける
- 保存先やファイルパスを意識しなくてよい
- サーバーへローカル直接接続できるWi-Fiへ接続すると写真が自動的にバックアップされる

### 3.3 想定利用規模

- 家族ユーザー: 2〜10名
- 登録端末: 2〜20台
- 管理ファイル数: 最大30万件を初期設計目標とする
- ストレージ容量: 接続HDDの容量に依存
- 同時利用: 家族数名による閲覧、再生、アップロード

## 4. 主要ユーザージャーニー

### 4.1 MVP: 外出先からファイルを一覧・詳細確認し、必要時にダウンロードする

1. ユーザーがAndroidアプリを開く
2. アプリがサーバーへ到達できないことを表示する
3. アプリがZeroTierの別アプリで接続を確認するよう案内する
4. ユーザーがZeroTierを接続し、KuraStorageアプリへ戻って再確認する
5. ログイン画面が表示される
6. ユーザーがログインする
7. ホーム画面からファイル一覧・詳細を確認し、必要な元ファイルをダウンロードする

### 4.2 MVP後: 許可された接続で写真を自動バックアップする

1. ユーザーがAndroidアプリで端末フォルダとサーバー保存先を設定する
2. ユーザーが外出中に写真や動画を追加する
3. 端末が、バックエンドへローカル直接接続できるWi-Fi、または許可済み外部Wi-Fi＋ZeroTierへ接続する
4. アプリのバックグラウンド処理がサーバー到達性を確認する
5. 新規・変更されたファイルだけを抽出する
6. サーバーへ自動アップロードする
7. 端末で削除されたファイルはサーバーから削除しない
8. アプリに最終バックアップ日時と結果を表示する

### 4.3 MVP後: 家族へファイルまたはフォルダを共有する

1. 所有者が共有したいファイルまたはフォルダを選択する
2. 共有する家族と権限を選択する
3. 対象ユーザーの「共有」画面に共有項目が表示される
4. フォルダを共有した場合、付与された権限は配下のファイルとフォルダへ継承される
5. ファイルを共有した場合、付与された権限はそのファイルだけに適用される
6. 対象ユーザーは付与された権限の範囲で閲覧または編集する

### 4.4 MVP後: テキストファイルを編集する

1. ユーザーがテキストファイルを開く
2. アプリ内のテキストエディターで内容を編集する
3. 「保存」を押す
4. バックエンドが権限と更新競合を確認する
5. 正常保存後、更新日時を画面へ反映する

### 4.5 MVP後: HDD上で外部変更されたファイルを扱う

1. HDD上でファイルが直接移動または削除される
2. バックエンドが変更検知または再スキャンで差異を発見する
3. HDDが正常に接続されていることを確認する
4. 実ファイルが存在しない項目を`MISSING`として記録する
5. アプリ上で「ファイルが見つかりません」と表示する
6. ユーザーが「一覧から削除」を選ぶと、関連する管理情報を削除する

### 4.6 MVP後: 任意の接続環境で表示品質を選んで写真・動画を閲覧する

1. ユーザーがローカル直接接続またはZeroTier経由でKuraStorageへ接続する
2. アプリが現在の接続環境に応じた初期画質を表示する
3. ユーザーが接続環境に関係なく、低画質、中画質、元画質のいずれかを選択する
4. バックエンドが対象品質のキャッシュを確認する
5. 写真の対象品質キャッシュがない場合、バックエンドが元画像から必要時に生成し、生成状態または対象品質画像を返す
6. 動画の対象品質キャッシュがない場合、バックエンドが非同期の変換ジョブを登録し、アプリが待機、バックグラウンド生成、元画質再生を選択できる状態を表示する
7. 低・中画質動画は変換完了後に利用可能とし、変換途中のデータは再生用として公開しない
8. アプリはユーザー操作なしに元画質へ自動フォールバックせず、元画質を選ぶ場合はファイルサイズまたは推定通信量を表示する
9. バックエンドは選択された品質のデータだけをクライアントへ送信する
10. ユーザーは閲覧中も品質を変更できる
11. 生成済みキャッシュは最終アクセスから24時間保持する

### 4.7 MVP後: 登録済み外部Wi-Fiで自動バックアップする

1. ユーザーが設定画面で現在接続中のWi-Fiを自動バックアップ対象として登録する
2. 端末が登録済み外部Wi-Fiへ接続する
3. アプリがZeroTier接続状態、TLS証明書とKuraStorageエンドポイント情報、ユーザー認証を確認する
4. 条件をすべて満たす場合だけ保留中のファイルを自動アップロードする
5. 未登録Wi-Fiまたはモバイル通信の場合、自動アップロードを開始しない
6. 手動閲覧や手動転送は、ZeroTier接続と認証が有効な場合に限り利用できる

---

### 4.8 管理CLIでユーザーを作成する

1. 管理者がRaspberry Piのローカルコンソールへログインする
2. `sudo KuraStorage-admin user create`を実行する
3. ユーザー名、表示名、ロール、初期パスワードを入力する
4. バックエンドの共通管理処理が、Argon2idの現在設定でパスワードをハッシュ化してUserを作成する
5. 操作結果を監査ログへ記録する
6. Androidアプリや通常のHTTP APIからユーザー作成は行えない

### 4.9 新しいAndroid端末で初回ログインする

1. 管理者が事前に管理CLIでUserを作成する
2. 利用者が新しいAndroid端末をKuraStorageへローカル直接接続できるネットワークへ接続する
3. Androidアプリが`LOCAL_DIRECT`を確認する
4. Androidアプリが端末内に有効な`deviceId`とRefresh Tokenがないことを確認し、初回登録を伴うログイン画面を表示する
5. 利用者がユーザー名とパスワードを入力する
6. Androidアプリが端末名とログイン資格情報をTLS通信でバックエンドへ送信する
7. バックエンドが接続経路、Userの状態とセキュリティロック、パスワードを確認する
8. バックエンドがDeviceとRefresh Sessionを作成し、`deviceId`とTokenを返す
9. Androidアプリが認証情報を安全に保存してログインを完了する

### 4.10 登録済み端末からZeroTierで利用する

1. 利用者がKuraStorage外のZeroTierアプリでNetworkへ接続する
2. AndroidアプリがZeroTier経路のKuraStorage HTTPSエンドポイントへの到達性とTLSを確認する
3. AndroidアプリがRefresh TokenでAccess Tokenを取得する
4. バックエンドがUser、Device、Refresh Sessionの有効性を確認する
5. 認証後もファイル・フォルダごとに権限を確認して操作を許可する
6. ZeroTier経由では新規Device登録を実行しない

### 4.11 紛失・侵害端末を失効する

1. 管理者が管理CLIでUserに関連するDevice一覧を確認する
2. 対象Deviceを指定して失効コマンドを実行する
3. バックエンドがDeviceを`REVOKED`へ変更する
4. 対象DeviceのRefresh Sessionをすべて失効する
5. 対象Deviceの未完了アップロードを停止する
6. 失効理由と実行OSユーザーを監査ログへ記録する
7. 管理者がZeroTier Centralまたは自己管理Controllerで対象Memberを別途失効する

## 5. 成功指標（KPI）

本プロダクトは家族向けの私的利用を主目的とするため、利用者数ではなく、MVPの基本操作、データ保護、実機での再現性を中心に評価する。自動バックアップ、共有、プレビュー等の指標は各機能を追加する際に定義する。

### 5.1 プライマリーKPI

- **基本操作完了率**: 初見の家族ユーザーが、説明なしで「ログイン」「一覧表示」「手動アップロード」「ダウンロード」「ゴミ箱移動・復元」を完了できる割合が90%以上
- **権限逸脱防止**: 権限のないユーザーが他ユーザーの非共有ファイルを閲覧・変更できる事象を0件とする
- **不完全ファイル公開防止**: アップロード未完了または破損状態のファイルが通常一覧に表示される件数を0件とする
- **誤保存防止**: HDD未マウント時にOSルートファイルシステムへファイルを保存する事象を0件とする
- **実機再現性**: Raspberry PiとAndroid実機でMVP E2Eを10回連続して完了できる

### 5.2 セカンダリーKPI

- 正常な接続環境では、アプリ起動からログイン済み一覧表示まで通常10秒以内を目標とする
- 1,000件以下のフォルダ一覧を通常2秒以内に表示する
- 大容量転送中もClientとServerのいずれもファイル全体をメモリへ保持しない
- 通信中断、Token失効、Device失効、HDD利用不可、復元競合を利用者が判別できる

## 6. 前提条件と制約

### 6.1 ネットワーク制約

- バックエンド、Webアプリ、DBは一般のインターネットへ直接公開しない
- ZeroTier経路で到達可能であっても、Access Tokenによるアプリ認証を必須とする
- Webブラウザ版では、ZeroTier接続はOSまたは別のZeroTierクライアントで事前に行う
- ZeroTier MemberからはKuraStorage HTTPSエンドポイントだけへ接続可能とし、SSH、PostgreSQL、SMB、他のLAN端末、他のZeroTier端末への通信を拒否する
- モバイル通信では自動バックアップを実行しない
- ローカル直接接続可能なWi-Fiでは、Android端末とKuraStorageサーバーが管理者設定の同一IPサブネットに属することを確認したうえで、ZeroTierを介さない基盤ネットワークへ通信をバインドし、TLS証明書・ホスト名検証とAPI到達確認が成功した場合に自動バックアップを許可する
- 登録済み外部Wi-Fiでは、ZeroTier接続、TLS検証、User・Device・Session認証がすべて成功した場合に限り自動バックアップを許可する
- 未登録Wi-Fiでは自動バックアップを実行しない
- Wi-Fi名やBSSIDの一致は接続経路の判定条件に使用せず、同一IPサブネットの確認もユーザー認証やTLS検証の代替にしない
- 新規Android Device登録は`LOCAL_DIRECT`からの初回ログイン時だけ許可する
- ZeroTier経由の新規Device登録要求は拒否する
- ローカル直接接続とZeroTier接続の両方が利用可能な場合は、既定ではローカル直接接続を優先する
- バックエンドの待ち受け状態だけでなく、リバースプロキシ、TLS、API応答を含むエンドツーエンドの到達性を確認する
- Phase 1のHTTPS固定ホスト名はGit管理外の`docs/environment-info.md`にある`NET-API-HOSTNAME`とし、LAN接続時とZeroTier接続時のURLを同じHostnameへ統一する。公開項目定義は`docs/environment-info.example.md`を参照する
- TLSはNginxで終端し、KuraStorage専用プライベートRoot CAが直接発行したサーバー証明書を使用する
- Androidアプリは`TLS-ROOT-CA-CERT-PATH`からRelease Buildへ供給したRoot CA公開証明書を同梱し、Network Security Configurationで`NET-API-HOSTNAME`の値だけに信頼を限定する
- Productionでは証明書検証またはホスト名検証を無効化できず、全証明書を許可するTrustManagerや常に成功するHostnameVerifierを使用しない
- Root CA秘密鍵はパスフレーズで保護し、証明書発行後はKuraStorageサーバー、Repository、Androidアプリ、配布物へ置かない
- Phase 1の証明書発行と更新は手動運用とし、常時稼働CAは導入しない。証明書数または更新頻度が増えた場合にstep-ca等へ移行可能な役割分離を維持する

### 6.2 ストレージ制約

- KuraStorage専用のストレージルートをHDD上に設定する
- KuraStorageで管理する実ファイルは、原則として専用ルート配下に保存する
- HDD上の実ファイルの存在、内容、フォルダ構造を正とする
- DBはファイル本体を保存せず、管理情報と索引を保存する
- HDDが未接続または未マウントの場合、ファイルを`MISSING`と判定しない

### 6.3 Android制約

- MVPの最低対応バージョンはAndroid 10（API Level 29、`minSdk 29`）とする
- MVPの転送は利用者がStorage Access Frameworkで選択した単一ファイルの手動操作とする
- MVPではRoom、WorkManager、常時Foreground Serviceを使用しない
- 以下のバックグラウンド再実行・自動バックアップ制約はMVP後の機能追加時に適用する
- アプリを最近のアプリ一覧から消した場合やOSがプロセスを終了した場合でも、予約済みバックグラウンド処理は再実行可能とする
- ユーザーがAndroid設定からアプリを強制停止した場合、自動実行できないことを許容する
- 秒単位の完全リアルタイム同期は保証しない
- 省電力性を優先し、イベント検知、保留キュー、OSスケジューラー、定期確認を組み合わせる

### 6.4 運用制約

- KuraStorage以外のツールが正式管理領域を自由に変更すると、DB索引との差異が生じ得る
- File Browserは管理・緊急アクセス用途として扱い、原則読み取り専用を推奨する
- Syncthingを併用する場合は、正式領域ではなく受け入れ用領域へ保存する

**ユーザーストーリー**

- [ ] `deviceId`だけでは認証済みとみなさない
- [ ] 認証失敗時にユーザーの存在有無を判別できる過度に詳細な情報を表示しない

**優先度**: P0（必須）

### 7.2.2 管理CLIによるユーザー・セキュリティ管理

**ユーザーストーリー**  
システム管理者として、高権限操作をネットワークへ公開せず家族ユーザーと端末を安全に管理するために、Raspberry Pi上の専用CLIを使用したい。

**受け入れ条件**

- [ ] 初期管理者作成はRaspberry Pi上の管理CLIからだけ実行できる
- [ ] 一般ユーザー・追加管理者の作成は管理CLIからだけ実行できる
- [ ] ユーザーの無効化・再有効化は管理CLIからだけ実行できる
- [ ] パスワード強制再設定は管理CLIからだけ実行できる
- [ ] アカウントのセキュリティロック解除は管理CLIからだけ実行できる
- [ ] Androidアプリと通常のHTTP APIには、上記高権限操作の管理画面・管理エンドポイントを提供しない
- [ ] 管理CLIはRaspberry Piのローカルコンソール上で、rootまたは明示的に許可されたOS管理者だけが実行できる
- [ ] CLIはDBを直接無秩序に更新せず、バックエンドの共通Application処理を呼び出す

**優先度**: P0（必須）

### 7.2.3 認証失敗・アカウントロック制御

**ユーザーストーリー**  
利用者として、パスワード総当たり攻撃を抑止しつつ、正しいパスワードでログインできた場合は過去の入力ミスを引き継がない認証制御を利用したい。

**受け入れ条件**

- [ ] 認証失敗回数はUser単位で管理し、IP単位およびDevice単位のロック・レート制限は行わない
- [ ] 同一Userに対するパスワード認証が15分以内に10回連続して失敗した場合、そのUserをセキュリティロックする
- [ ] 10回へ達する前に正常ログインした場合、対象Userの認証失敗回数を0へリセットし、15分の計測期間を初期化する
- [ ] セキュリティロックは時間経過では自動解除しない
- [ ] セキュリティロックはRaspberry Pi上の管理CLIからだけ解除できる
- [ ] セキュリティロック時は対象UserのRefresh Sessionをすべて失効し、新規ログインとToken更新を拒否する
- [ ] 管理CLIでロックを解除した場合、認証失敗回数、計測期間、ロック情報を初期化する
- [ ] 存在しないユーザー名への失敗を特定Userのカウンターへ加算せず、ユーザーの存在有無を判別できる情報も返さない
- [ ] 失敗回数10回と計測期間15分は初期値とし、サーバー設定で変更可能とする
- [ ] 1ユーザーあたりの有効なDevice上限は初期値10台とし、`REVOKED`端末は上限に含めず、設定で変更可能とする
- [ ] ログイン成功・失敗、セキュリティロック、管理CLI解除を監査ログへ記録する

**優先度**: P0（必須）

### 7.2.4 パスワード保存

**ユーザーストーリー**  
利用者として、DBやバックアップが漏えいした場合でも平文パスワードが取得されず、オフライン解析に対して十分な計算コストを持つ方式で認証情報を保護したい。

**受け入れ条件**

- [ ] パスワードはArgon2id v1.3（`v=19`）でハッシュ化する
- [ ] MVP初期パラメータはメモリ19MiB、反復2回、並列度1とする
- [ ] パスワード設定または再設定ごとに、暗号学的に安全な16バイトのランダムSaltを生成する
- [ ] DBにはアルゴリズム、バージョン、パラメータ、Salt、ハッシュ値を含む自己記述形式の`passwordHash`だけを保存する
- [ ] 平文パスワード、復号可能な暗号文、MD5、SHA-1、SHA-256、SHA-512等の単純ハッシュを保存しない
- [ ] 同じパスワードを複数Userへ設定しても、異なるSaltにより異なる`passwordHash`が生成される
- [ ] ハッシュ生成・検証には保守されているArgon2idライブラリを使用し、独自暗号実装を行わない
- [ ] 正常ログイン時に保存済みハッシュパラメータが現在の基準より弱い場合、現在の設定で再ハッシュして更新する
- [ ] Raspberry Pi 4 Model B（8GB RAM）で通常1秒未満となることを実測し、性能に余裕があればパラメータを強化できる
- [ ] メモリ19MiB、反復2回、並列度1より弱い設定へ変更しない
- [ ] パスワード、Salt、ハッシュ値をログ、監査ログ、例外メッセージ、APIレスポンスへ出力しない

**優先度**: P0（必須）

**受け入れ条件**

- [ ] DBには実ファイルではなく相対パス、所有者、サイズ、種類、日時、状態などを保存する
- [ ] HDD上のファイル存在と内容を正とする
- [ ] ファイル一覧は主にDB索引から取得する
- [ ] 実際に開く際はHDD上の存在を確認する
- [ ] HDD変更をイベント検知または定期再スキャンでDBへ反映する

**優先度**: P0（必須）

### 7.10.4 MVP後: `MISSING`状態

**ユーザーストーリー**  
利用者として、HDD上から失われたファイルと一時的なストレージ障害を区別したい。

**受け入れ条件**

- [ ] HDDが正常な状態で実ファイルが存在しない場合だけ`MISSING`候補とする
- [ ] 再確認後も存在しない場合に`MISSING`へ確定する
- [ ] `MISSING`項目を通常ファイルと区別して表示する
- [ ] `MISSING`項目に「再確認」を提供する
- [ ] `MISSING`項目に「一覧から削除」を提供する
- [ ] 「一覧から削除」ではHDD操作を行わず、DB上の管理情報と関連情報を削除する
- [ ] HDDが利用不可の場合は`MISSING`へ変更しない

**優先度**: P0（必須）

## 7.11 MVP後: 派生データとキャッシュ管理

### 7.11.1 必要時生成

**ユーザーストーリー**  
利用者として、HDD容量を不必要に消費せず通信量を抑えるために、低・中画質データを必要なときだけ生成してほしい。

**受け入れ条件**

- [ ] 元ファイルのアップロード時に低画質版と中画質版を一律で永久保存しない
- [ ] 写真・動画の低画質または中画質が要求されたときに対象品質のデータを生成する
- [ ] 写真は要求処理内で短時間生成を試み、設定した待機時間を超える場合は生成状態を返してバックグラウンドで継続する
- [ ] 動画は変換ジョブをキューへ登録し、API要求とは分離されたWorkerで全体を生成する
- [ ] 動画派生データは全体の生成と検証が完了した後にだけ`READY`として公開する
- [ ] 動画変換の進捗、キュー待ち、完了、失敗をアプリから確認できる
- [ ] 変換中にユーザーが画面を離れても、ジョブが完了または失敗するまで継続する
- [ ] 低・中画質が未準備の場合、ユーザーの明示操作なしに元画質へ自動フォールバックしない
- [ ] 同一ファイル、同一元バージョン、同一品質の生成処理を同時に重複実行しない
- [ ] 現在の元ファイルバージョン、対象品質、変換設定に一致するキャッシュが存在する場合は再生成せず再利用する
- [ ] ファイル名の変更または移動はキャッシュ識別条件に含めず、内容が同一である限り既存キャッシュを再利用する
- [ ] 元ファイルの内容が更新された場合はファイルバージョンを更新し、旧バージョンから生成されたキャッシュを使用しない
- [ ] 使用されなくなった旧バージョンのキャッシュは、通常の保持期限または容量超過時の清掃によって削除する
- [ ] サムネイルは一覧表示用として通常の低・中画質キャッシュと分離して管理する
- [ ] 写真・動画のサムネイルは元ファイルが存在する間は全件保持し、容量上限や24時間TTLを適用しない
- [ ] 元ファイルの内容更新時はサムネイルを再生成し、名前変更または移動だけでは既存サムネイルを再利用する

**優先度**: P0（必須）

### 7.11.2 保持期限と容量上限

**ユーザーストーリー**  
管理者として、キャッシュによるHDD使用量を一定範囲に抑えるために、自動清掃を行いたい。

**受け入れ条件**

- [ ] 低・中画質キャッシュは最終アクセスから24時間保持する
- [ ] キャッシュ利用時に最終アクセス日時を更新する
- [ ] 通常の低・中画質キャッシュ合計容量の上限を10GBとする
- [ ] 合計容量が10GBを超えた場合、最終アクセスが古い順に削除する
- [ ] 容量超過による清掃は、合計容量が6GB以下になるまで継続する
- [ ] 生成中または現在利用中のキャッシュを削除しない
- [ ] キャッシュ削除失敗時は元ファイルへ影響を与えず、後続の清掃で再試行する
- [ ] キャッシュは消失しても元ファイルから再生成できる
- [ ] 10GB上限と6GBまでのLRU清掃対象は低・中画質キャッシュだけとし、写真・動画サムネイルは対象外とする

**優先度**: P0（必須）

### 7.11.3 元ファイル削除・欠損時の連動

**ユーザーストーリー**  
利用者として、元ファイルを削除または見失った際に、不要な派生データを残さず整理したい。

**受け入れ条件**

- [ ] アプリ内から`ACTIVE`ファイルを削除した場合、元ファイルをゴミ箱へ移動し、紐づく低・中画質キャッシュを削除する
- [ ] ゴミ箱内の一覧表示に使用するサムネイルは元ファイルの完全削除まで保持する
- [ ] ゴミ箱から復元した場合、保持中のサムネイルを再利用し、低・中画質キャッシュは必要時に再生成する
- [ ] 元ファイルがHDD上から直接削除され`MISSING`になった場合、残存キャッシュを正式ファイルの代替として提供しない
- [ ] `MISSING`項目を「一覧から削除」した場合、管理情報、共有情報、同期情報、サムネイル、低・中画質キャッシュを削除する
- [ ] 元ファイルが同じ場所へ復旧した場合、旧キャッシュを無効化し、必要時に再生成する

**優先度**: P0（必須）

## 9. 非機能要件

### 9.1 セキュリティ

- バックエンドとWebアプリを一般のインターネットへ直接公開しない
- KuraStorage APIは許可したローカルネットワーク範囲およびKuraStorage用ZeroTier Networkだけから受け付ける
- ZeroTier MemberからはKuraStorage HTTPSだけへ到達可能とし、SSH、PostgreSQL、SMB、他端末への通信を拒否する
- ZeroTier Network内でもTLSを必須とし、証明書とホスト名を検証する
- 保護APIは既定で認証必須とし、匿名利用可能なAPIを明示的に限定する
- ZeroTier接続だけでは認証済みとみなさない
- Access Token認証後も、すべてのファイル操作でサーバー側が対象リソースの権限を確認する
- クライアントから送信された`userId`や`deviceId`を認証・認可の根拠としてそのまま信用しない
- パスワードはArgon2id v1.3（`v=19`）でハッシュ化し、MVP初期値をメモリ19MiB、反復2回、並列度1とする
- パスワード設定ごとに暗号学的に安全な16バイトのランダムSaltを生成する
- DBにはアルゴリズム、バージョン、パラメータ、Salt、ハッシュ値を含む自己記述形式だけを保存する
- 平文、復号可能な暗号文、MD5、SHA系の単純ハッシュを保存せず、独自暗号実装を行わない
- 正常ログイン時に保存済みパラメータが現在の基準より弱ければ、現在の設定で再ハッシュする
- Access Tokenを短寿命とし、Refresh Tokenを端末単位で管理、ローテーション、ハッシュ保存する
- 使用済みRefresh Tokenの再利用を検知し、関連するSession系列を失効できる
- 新規Device登録を`LOCAL_DIRECT`からだけ許可し、ZeroTier経由では拒否する
- User単位でログイン失敗を管理し、15分以内に10回連続失敗した場合は管理CLI解除が必要なセキュリティロックとする
- 正常ログイン時は対象Userの失敗回数を0へリセットし、15分の計測期間を初期化する
- 高権限管理操作をAndroid管理画面や通常HTTP APIから提供しない
- 管理CLIはRaspberry Piのローカルコンソール上で、rootまたは許可されたOS管理者だけが実行できる
- APIプロセスをroot権限で動かさない
- クライアントから任意のOSパスを受け取らない
- ファイルIDから保存場所を解決する
- KuraStorage専用ルートだけを読み書き可能とする
- パストラバーサルとシンボリックリンク経由の領域外アクセスを防止する
- アップロードファイル名、サイズ、実ファイル形式を検証する
- 認証、端末登録、失効、管理CLI、重要なファイル操作を監査記録する
- ZeroTier切断時は進行中の閲覧、アップロード、ダウンロードを停止または安全に一時停止する

### 9.2 パフォーマンス

ローカル直接接続、正常なHDD、家族利用規模を前提として次を目標とする。

- ホーム画面の主要情報を通常2秒以内に表示する
- 1,000件以下のフォルダ一覧を通常2秒以内に表示する
- メタデータ30万件規模で、一般的な名前検索を通常2秒以内に返す
- キャッシュ済み写真のプレビューを通常2秒以内に開始する
- 未生成の写真プレビューでは通常1秒以内に生成状態を返す
- キャッシュ済み動画・音声を通常3秒以内に再生開始する
- 未生成の動画派生データでは通常1秒以内に変換待ちまたは生成状態を返す
- 動画変換はHTTP要求処理から分離し、長時間変換によってAPI要求をタイムアウトさせない
- 基準サーバーはRaspberry Pi 4 Model B（8GB RAM）とし、動画変換の同時実行数は1とする
- 動画変換完了時間は動画長、解像度、コーデック、変換品質、Raspberry Pi 4 Model Bの性能に依存するため固定値を保証しない
- ファイル一覧表示のために毎回HDD全体をスキャンしない
- 同一HDD内の名前変更・移動では、不必要な全体コピーを行わない

### 9.3 信頼性

- アップロードは一時領域への保存完了後に正式配置する
- 通信中断時に不完全ファイルを正式ファイルとして扱わない
- HDD操作とDB更新の一方だけが成功した場合を検出する
- HDD未接続時にRaspberry Pi本体ストレージへ誤保存しない
- HDD利用不可と個別ファイル`MISSING`を区別する
- バックアップ処理の再試行が可能である
- 端末側削除によってサーバー側ファイルを削除しない
- 派生キャッシュの生成、削除、清掃に失敗しても元ファイルを変更しない
- 生成途中または検証未完了の動画派生データを閲覧可能な状態にしない
- 動画変換ジョブはアプリ画面の終了や一時的なクライアント切断と独立して継続・状態確認できる
- 10GB超過時のキャッシュ清掃は6GB以下になるまで実行し、生成中・利用中データを除外する
- KuraStorage自体とは別に、重要ファイルの二次バックアップを運用できる構成とする

### 9.4 省電力

- 常時ポーリングを行わない
- 常時Foreground Serviceを標準使用しない
- OSが提供する変更情報とバックグラウンドスケジューラーを優先する
- 毎回全ファイルのチェックサムを計算しない
- 自動バックアップ条件を満たさない接続では短時間の判定だけで処理を終了する
- 低バッテリー時にバックアップを停止できる
- 大量初回バックアップを充電中だけに制限できる
- モバイル通信では自動バックアップを起動しない
- 未登録Wi-Fiでは自動バックアップを起動しない
- プレビューキャッシュは24時間再利用し、同じデータの再生成回数を抑える

### 9.5 保守性

- AndroidとWebが同じバックエンドAPIを利用できる
- フロントエンド固有の状態をバックエンドへ混在させない
- 実ファイル、管理DB、設定、ログを分離する
- ストレージルートのマウント先を変更しても、相対パス管理により大規模なDB修正を避ける
- ファイルスキャンと索引更新を再実行可能にする

### 9.6 可観測性

- ログイン成功・失敗、アカウントセキュリティロック・解除を記録する
- 管理CLIのコマンド、対象、結果、実行OSユーザー、理由を記録する
- Refresh Token再利用検知を記録する
- 管理者が最近のエラーと監査操作を確認できる
- 個人情報や認証秘密情報をログへ出力しない

## 10. データ要件

### 10.1 HDDへ保存するもの

- ユーザーが保存する実ファイル

- ゴミ箱内ファイル
- サムネイル
- 必要に応じて過去バージョン
- 一覧用サムネイル
- 必要時に生成した低・中画質の一時キャッシュ

### 10.2 DBへ保存するもの

- Userとロール、状態、Argon2id自己記述形式の`passwordHash`、認証失敗・ロック情報
- Deviceと所有User、端末名、状態、最終アクセス、失効情報
- Refresh SessionとDeviceの関連、Tokenハッシュ、Token系列、有効期限、失効情報
- 認証試行、User単位の失敗回数、計測期間、セキュリティロックの記録
- 管理CLI、端末登録、失効などの監査ログ
- ファイルIDと相対パス
- ファイル名、種類、サイズ、日時
- 所有者
- 共有相手と権限
- ファイル状態
- 自動バックアップルール
- 端末側ファイルとサーバー側ファイルの対応
- バックアップ結果
- 最近使用したファイル
- 将来的なタグ、お気に入り、履歴
- 派生キャッシュの種類、元ファイルバージョン、サイズ、最終アクセス日時、有効期限、生成状態

### 10.3 Android端末内へ保存するもの

- バックエンドが発行した`deviceId`
- Device表示名
- 選択された端末フォルダの永続アクセス情報
- 自動バックアップルール
- 端末側ファイルの差分索引と保留キュー
- 登録済みWi-FiのSSID、任意のBSSID、表示名、従量制設定、許可状態
- 通信種別ごとの初期画質
- Refresh Token


Wi-Fi情報は端末ごとの通信ポリシーとして扱い、サーバー認証情報としては使用しない。

### 10.4 ファイル状態

最低限、次の状態を管理する。

- `ACTIVE`: HDD上に実ファイルが存在し、通常利用可能
- `MISSING_CANDIDATE`: HDDは正常だが、初回確認で実ファイルが見つからない
- `MISSING`: 再確認後も実ファイルが存在しない
- `TRASHED`: ゴミ箱へ移動済み。復元に必要な`FileEntry`と元の場所の情報を保持する
- `UPLOADING`: アップロード処理中
- `ERROR`: 整合性または処理エラー

`DELETED`状態は設けない。ゴミ箱への移動中は`TRASHED`として管理するが、ユーザーによる完全削除または30日経過後の自動完全削除が完了した時点で、実ファイルと`FileEntry`を含む関連管理情報を削除する。削除後のファイルを通常のファイル索引へ残さず、監査が必要な場合はファイル索引とは独立した監査ログへ削除イベントだけを記録する。

### 10.5 物理構成の論理要件

KuraStorage専用ルート配下に、少なくとも次の用途を分離する。

```text
KuraStorage/
├─ private/       ユーザー個人領域
├─ shared/        家族共有領域
├─ incoming/      外部取り込み・受け入れ領域
├─ trash/         ゴミ箱
├─ thumbnails/    一覧用サムネイル
├─ cache/         低・中画質の一時キャッシュ
├─ versions/      過去バージョン
└─ upload-temp/   アップロード途中
```

実際のディレクトリ名や配置はアーキテクチャ設計書で確定する。

## 11. エラー処理要件

### 11.1 接続・認証・端末エラー

- ZeroTier未接続
- ZeroTier接続失敗
- サーバー到達不可
- TLS証明書またはホスト名の検証失敗
- 未登録端末
- 新規端末登録にローカル直接接続が必要
- Device登録失敗
- Device失効
- Refresh Session失効
- Refresh Token再利用検知
- アカウントのセキュリティロック

それぞれ原因と再試行、ローカル接続、再ログイン、管理者への連絡など可能な次の操作を表示する。ユーザー名が存在するかどうかを推測できる詳細は返さない。

### 11.2 ストレージエラー

- HDD未接続
- マウント失敗
- 読み取り専用
- 空き容量不足
- ファイル不存在

HDD全体の問題と個別ファイルの問題を区別する。

### 11.3 ファイル操作エラー

- 権限不足
- 同名ファイル競合
- 編集競合
- アップロード中断
- ダウンロード中断
- 対応していない形式
- 対象操作に設定された処理可能サイズ上限の超過

アップロードまたはダウンロードは、ファイルサイズが大きいことだけを理由に一律拒否しない。保存先の空き容量不足はストレージエラーとして扱い、テキスト編集など特定機能のサイズ上限を超えた場合は、適用された上限値とダウンロードなどの代替操作を表示する。再試行、別名保存、処理方法の変更、キャンセルなど、可能な次の操作を提示する。

### 11.4 プレビュー・キャッシュエラー

- 元ファイル`MISSING`
- 派生データ生成失敗
- 対応していない変換形式
- 変換処理待ち
- キャッシュ容量不足
- キャッシュ削除失敗

元ファイルへの影響を避け、元画質利用、再試行、待機、管理情報削除など可能な次の操作を提示する。

### 11.5 バックアップエラー

- 自動バックアップ許可条件外
- バッテリー条件未達
- サーバーまたはHDD利用不可
- 端末側アクセス権消失
- ファイル読み取り失敗
- アップロード失敗

自動で再試行可能なものと、ユーザー対応が必要なものを区別する。

## 13. リスクと対策方針

### 13.1 HDDが唯一の保存先になるリスク

**リスク**: HDD故障時にファイルを失う。  
**方針**: KuraStorageの機能とは別に、第二のHDDまたは別媒体へのバックアップ方針を用意する。

### 13.2 HDDとDB索引の不整合

**リスク**: SSH、File Browser、Syncthingなどによる直接変更で索引がずれる。  
**方針**: MVPは正式領域への書き込みをKuraStorageへ集約し、`FileOperation`対象の途中状態だけを復旧する。外部変更検知と定期再スキャンはMVP後に追加する。

### 13.3 MVP後: Androidのバックグラウンド制限

**リスク**: メーカー独自の省電力制御や強制停止により処理が遅延する。  
**方針**: 常駐を前提とせず、変更検知、保留キュー、OSスケジューラー、アプリ起動時確認を組み合わせる。

### 13.4 ZeroTier接続の複雑さ

**リスク**: 初回設定、Member未認可、または他のVPNアプリとの競合で接続できない。  
**方針**: 初回導入フロー、状態表示、再試行、設定確認を分かりやすく提供する。

### 13.5 MVP後: 大容量動画による負荷

**リスク**: Raspberry Pi、HDD、Wi-Fi、端末バッテリーに負荷がかかる。  
**方針**: Range Request、分割転送、再開、バックアップ条件、転送キューを採用する。

### 13.6 MVP後: 低・中画質動画の変換負荷

**リスク**: Raspberry Pi上での動画変換が再生速度に追いつかず、CPU使用率や消費電力が上がる。  
**方針**: 写真は短時間の必要時生成、動画はAPI要求から分離した非同期変換ジョブとする。Raspberry Pi 4 Model B（8GB RAM）では動画変換同時数を1に制限し、低画質720p・中画質1080pの完成済みMP4（H.264＋AAC）を生成する。進捗表示、元画質再生の明示選択、24時間キャッシュ、10GB上限と6GBまでのLRU清掃を採用する。変換途中の動画を公開せず、画質値は初期値として実機計測後に設定変更できるようにする。

### 13.7 MVP後: 登録Wi-Fiの誤認

**リスク**: 同一SSIDの偽アクセスポイントやテザリングを、許可済みWi-Fiとして認識する可能性がある。  
**方針**: Wi-Fi一致は実行候補の判定に限定し、外部Wi-FiではZeroTier、TLS、サーバー識別、ユーザー認証を必須にする。

### 13.8 MVP後: 共有権限の設定ミス

**リスク**: 非公開のファイルまたはフォルダを意図せず家族へ共有する。  
**方針**: 初期状態を非共有とし、共有対象の種類、共有相手、権限、フォルダ共有時の継承範囲を確認画面で明示する。

---

### 13.9 端末資格情報の盗難・端末紛失

**リスク**: 紛失端末に残るRefresh Tokenが悪用される、またはZeroTier Member認可が残りネットワーク到達性が維持される。  
**方針**: Refresh TokenをAndroid Keystoreで保護し、管理CLIのDevice失効で関連Sessionを失効する。それとは別に、管理者がZeroTier管理面で対象Memberを失効し、アプリ認証とネットワーク参加の両方を無効化する。

### 13.10 自動端末登録の悪用

**方針**: Device登録を`LOCAL_DIRECT`に限定し、TLS検証、User状態、アカウントのセキュリティロック、端末数上限を確認する。端末登録を監査ログへ記録し、不審なDeviceを管理CLIから失効できるようにする。



## 14. リリース判定基準

MVPは、以下をすべて満たした場合に家庭内での初期利用を開始できる。

- [ ] ローカル直接接続でAndroidアプリから接続、ログインできる
- [ ] ZeroTier経由で接続、ログインできる
- [ ] Local Direct以外から新しいDeviceを登録できない
- [ ] 登録済みDeviceでToken更新、Logout、管理CLIからの失効が機能する
- [ ] Argon2id、Access Token 15分、Refresh Token 24時間・Rotation・再利用検知が設計どおり機能する
- [ ] ユーザーごとの個人領域が分離され、他Userの項目を操作できない
- [ ] 一覧、詳細、フォルダ作成をAndroidから実行できる
- [ ] SAFで選択した単一ファイルを、Client・Serverとも全体をメモリへ保持せずStreaming Uploadできる
- [ ] 通信中断時は同じIdempotency Keyでファイル全体を再試行でき、不完全ファイルを公開しない
- [ ] 元ファイルをHTTP RangeでSAFの選択先へDownloadできる
- [ ] ファイル・フォルダをゴミ箱へ移動し、元の場所へ復元できる
- [ ] 復元先の同名項目を上書きせず、競合エラーを返す
- [ ] HDD未マウント時にOSルートへ誤保存せず、途中操作を`FileOperation`から安全に復旧できる
- [ ] Server・Androidの必須CI、結合テスト、Raspberry Pi・Android実機E2Eが成功する
- [ ] Release APK、Server配置、Migration、更新、ロールバックの再現可能な手順がある
